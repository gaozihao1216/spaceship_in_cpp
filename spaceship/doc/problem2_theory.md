# Problem 2 Theory

## Goal

Problem 2 should be defined as a graph search problem built on top of the Problem 1 transfer oracle, before any production search code is written.

## Mathematical Objects

### Planet Set

Let

`P = {Earth, Mars, ...}`

be the set of candidate planets or encounter bodies.

### Encounter State

An encounter state should include at least:

- current planet
- current time
- accumulated delta-v
- path history
- branch metadata

Depending on fidelity, it may also need:

- incoming transfer geometry
- outgoing transfer geometry
- last transfer orbit parameters
- a compact incoming velocity or branch class state

### Transfer Edge

Problem 1 supplies a transfer edge from planet `A` to planet `B`.

Each edge should include:

- departure planet `A`
- target planet `B`
- departure anomaly or departure time
- arrival anomaly or target time
- transfer time of flight
- transfer revolution `k`
- target revolution `q`
- encounter alpha
- transfer orbit parameters
- residual and validity diagnostics
- estimated delta-v contribution

### Problem 1 Solver As Oracle

Problem 2 should not re-solve all orbital geometry internally.

It should call a Problem 1 oracle:

1. try Route B cached Hessian query
2. if valid and branch complete, use Route B result
3. else fallback to Route A
4. if Route A invalid, no edge

Problem 2 should not depend on online Hessian Route B.

## Abstract Search Graph

### Node

A graph node or label contains:

- current planet id
- current time or time bucket
- accumulated cost
- path sequence
- branch metadata

### Directed Edge

A directed edge is a feasible transfer from the current planet to the next planet, generated by the Problem 1 oracle.

### Cost

Candidate cost models include:

- delta-v
- time penalty
- mission objective penalty
- risk or invalidity penalty

### Feasibility

Feasibility filters may include:

- time window
- max total flight time
- max revolutions
- max encounters
- minimum transfer time
- maximum transfer time
- per-planet encounter constraints

## Search Algorithm Choice

### BFS

If edge cost is effectively uniform, BFS can be acceptable.

### Dijkstra

If cost is weighted by delta-v or time, the graph is weighted and Dijkstra is the natural baseline.

### A*

If there is a target planet and a meaningful lower-bound heuristic, A* should be considered.

### Dynamic Programming / Label Setting

If state includes continuous time, the search likely needs discretization or a label-setting formulation.

### Multi-Label Shortest Path

If one planet-time state can hold multiple Pareto-optimal labels, then Problem 2 is not a single-label shortest path problem. It becomes a multi-label shortest path problem.

This is why Problem 2 is not automatically “just BFS”. The state may depend on:

- time
- delta-v
- incoming geometry
- branch identity
- path constraints

## Dominance Pruning

A first dominance rule draft is:

Label `L1` dominates `L2` if:

- same current planet
- similar current time bucket
- `L1.accumulated_delta_v <= L2.accumulated_delta_v`
- `L1.elapsed_time <= L2.elapsed_time`
- `L1.remaining_flexibility` is no worse
- and at least one inequality is strict

But this rule is only safe if incoming geometry does not materially affect the next transfer cost.

If incoming geometry changes downstream feasibility or delta-v, then planet/time labels cannot be merged solely by planet and time. In that case the state must retain incoming branch class or velocity-state information.

## Problem 1 Oracle Strategy

Future `query_transfer(A, B, departure_state)` strategy:

1. try Route B cached Hessian query
2. if valid and branch complete:
   use Route B result
3. else:
   fallback Route A
4. if Route A invalid:
   no edge

This keeps Problem 2 dependent on a stable oracle boundary, rather than embedding Route B online Hessian work into the search itself.

## Minimal Implementation Roadmap

### Version 0

- fixed planet list
- fixed max depth
- fixed discrete launch time grid
- each edge uses Problem 1 Route A
- goal is only to find feasible paths

### Version 1

- add Route B cached query
- fallback Route A
- add cost such as total time or approximate delta-v

### Version 2

- multi-label Dijkstra
- dominance pruning
- time window constraints

### Version 3

- more realistic delta-v
- incoming/outgoing velocity matching
- gravity assist or encounter continuity constraints

## Open Questions

1. What is the true Problem 2 objective?
   - minimize delta-v?
   - minimize time?
   - maximize reachable planets?
   - find any feasible path?

2. Are impulsive corrections allowed at intermediate encounters?

3. Must velocity be continuous at each encounter, or is geometric reachability enough?

4. Is waiting time allowed?

5. Is repeated visitation of the same planet allowed?

6. Are `max_transfer_revolution` and `max_target_revolution` globally constrained?

7. Is the output a single optimum, or a Pareto frontier?
